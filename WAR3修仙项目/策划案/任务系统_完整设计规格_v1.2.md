## 第一部分：架构层

### 1. 任务系统定位

任务系统是**编排层**。它只做四件事：

1. 接收战斗/背包/对话/修炼/突破等领域系统的事实变化
2. 按任务定义推进当前步骤/进度/状态
3. 调用导航/教学/演出/奖励/解锁/存档接口
4. 向 HUD/任务面板/NPC 标记提供任务视图

业务系统不感知"某任务是否存在"，只广播 `alchemy.recipe_completed`、`realm.changed` 等事实。任务、成就、图鉴、埋点、世界事件共享同一批事件源。

```
战斗/背包/对话/采集/炼丹/修炼/突破/宗门
                  ↓ 领域事件 (se.emit)
       玩家事实层（Fact / Anchor / DepKey）
                  ↓ 增量刷新
     QuestRuntime（状态 + 当前Step + 条件树 + 进度）
         ↓                 ↓                 ↓
      任务UI             奖励/解锁          存档/服务器同步
```

### 4. 任务 + 步骤 双层模型

旧框架条件树只判"是否满足"，新手教学还要判"现在该教什么"，因此必须新增显式步骤机。

```
QuestDef
  ├─ taskId / category / scope / trigger / repeat / rewards
  └─ StepDef[]
       ├─ stepId / title / objectiveMode
       ├─ enterCondition / objectives
       ├─ onEnter（导航/教学/传送/演出）
       ├─ onComplete（对白/解锁/下一步骤）
       └─ deliveryRule / failRule

QuestInstance
  ├─ status / currentStepId / stepProgress
  ├─ claimed / instanceId / baseline facts
  └─ updatedAt / version
```

例：1.3 炼丹任务——到灵植园 → 药童对白 → 采集凝露草 → 周文扬对白 → 炼 3 颗引气丹 → 服 1 颗 → 调息完成。这些不能平铺进一个 `all` 条件树，否则玩家可能在教学前就满足后续条件。

### 4.1 层级裁定：二级（任务 + 步骤），不做三级子任务【本版裁定】

设计前提里"每个大任务内穿插多个小任务"与模型里的 `QuestDef + StepDef` 存在歧义。**裁定：采用二级模型——`QuestDef`（任务）+ `StepDef[]`（步骤），不引入独立的"子任务"实体。** 理由：

- 存档：子任务若是独立实体，需各自 status/进度/领奖记录，存档字段翻倍；步骤只需 `currentStepId + stepProgress`，轻。
- 奖励：新手线是"整条完成给一次奖励 + 少量步骤即时反馈"，不需要子任务级独立领奖。
- UI：任务栏只显示"当前任务 + 当前步骤目标"，三级会让追踪列表层级过深。
- 前置：跨任务前置用 `completed_quest(taskId)`；步骤间顺序用 `nextStep`——两套已够，无需子任务前置。

术语统一：**"大任务节点"＝ Quest；"小任务/教学环节"＝ Step。** 全文"小任务"一律指 Step。若未来出现"一个 NPC 一次给多个可独立接取/放弃的委托"，那是多个 Quest，不是子任务。

### 4.2 StepDef 字段定义【本版补：原文只列字段名无语义】

| 字段 | 类型 | 取值 / 行为 |
|---|---|---|
| `stepId` | int | 任务内唯一，从 1 递增即为默认顺序 |
| `title` | string | 任务栏当前目标文案 |
| `objectiveMode` | enum | `SEQUENTIAL`（步骤内多目标须按序）/ `ALL`（全部满足即可，无序）/ `ANY`（任一满足）。**默认 `ALL`**；新手需强制手感顺序的用 `SEQUENTIAL` |
| `enterCondition` | RuleTree? | 进入本步骤的前置（缺省＝上一步 onComplete 即进入） |
| `objectives` | Predicate[] | 本步骤完成条件（复用 §5 谓词），满足即步骤达成。每个目标须声明 `countMode`：`CUMULATIVE`（累计型，跨步骤总量）或 `STEP_LOCAL`（本步骤起算型，见下方 stepBaselines）|
| `onEnter` | Action[] | 进入时执行的动作序列，每个 Action 带执行语义（可重放/一次性），见下方定义 |
| `onComplete` | Action[] | 完成时执行：对白/解锁/即时反馈。**不在此发正式奖励**（正式奖励在任务 claim，走 §6.3） |
| `deliveryRule` | struct? | 交付规则，见下。缺省＝步骤达成即完成，无需交付 |
| `failRule` | struct? | 失败/超时规则，见下。缺省＝不可失败 |

**`deliveryRule`（关键，绑定 §6.3 领奖原子顺序）**：`{ npcId, dialogueNodeId, mustInteract:bool }`。`mustInteract=true` 时即使 `objectives` 已满足，任务仍停在 `COMPLETABLE`，必须玩家到 `npcId` 触发 `dialogueNodeId` 才 `claim`。这正是修复迁移 bug⑨"交付可被绕过"的机制——交付是显式步骤条件，不靠 UI 约束。新手线凡"回 X 处交任务"的末步 `mustInteract=true`。

**`failRule`**：`{ kind: "TIMEOUT"|"DEATH"|"LEAVE_INSTANCE", onFail: "RESET_STEP"|"RESET_QUEST"|"RETRY_IN_PLACE" }`。新手线默认**不启用失败**；仅 1.8 心魔战用 `{DEATH, RETRY_IN_PLACE}`（原地重试、不重复扣材料）。限时/护送失败留后续扩展，V1 不做。

**`onEnter` Action 执行语义【本版补：防重连重复传送/演出/解锁】**：`onEnter` 会在断线重连、切场景恢复时按当前步骤重放，因此每个 Action 必须标类型：

| Action 类型 | 举例 | 重连行为 |
|---|---|---|
| `REPLAYABLE` 可重放 | 导航寻路、教学高亮、UI 提示、镜头指向 | 每次进入/重连都重放，无副作用 |
| `ONCE` 一次性 | 传送、功能解锁、剧情演出、发服饰/秘籍 | 首次执行后写 `doneActionIds[]`；重连时若已在集合内则跳过，**不重复传送/演出/解锁** |
| ❌ 禁止入 onEnter | 正式任务奖励 | 只能在 `tryClaim` 走 §6.3 原子顺序结算，绝不放 onEnter/onComplete |

**`stepBaselines` 步骤基线【本版补：基线细化到步骤首次进入】**：任务级 `baseline_facts` 只解决"接取时刻起算"，不够——玩家可能在前置步骤里已顺手满足了后续步骤的事实。规则：

- `STEP_LOCAL` 目标：在**该步骤首次激活**时，把当前锚点值写入 `stepBaselines[stepId][depKey]`，之后进度 = 当前值 − 该步骤基线。
- `CUMULATIVE` 目标：不写步骤基线，用任务级 `baseline_facts`（接取起算）。
- **重连重放 `onEnter` 时不得重置 `stepBaselines`**（基线只在步骤首次激活写一次，存档持久化）。
- 默认：新手教学的计数目标（杀 4 只、炼 3 颗）用 `STEP_LOCAL`，保证"这一步现做现算"；跨任务的持有量/境界阈值用 `CUMULATIVE`。

### 4.3 状态机与流转规则【本版补：原文只写字段无规则】

**生命周期（权威、存档）**：`NOT_ACCEPTED → ACCEPTED → COMPLETABLE → COMPLETED`，旁支 `ABANDONED`/`FAILED`（新手线不用）。**视图态（派生、不存档）**：`HIDDEN/VISIBLE/ACCEPTABLE/ACTIVE/COMPLETABLE/COMPLETED`，由（生命周期 + show/accept/complete 三态树）推导，沿用旧 `computeViewState`。领奖 `claimed` 与 `COMPLETED` 分离。

流转：接取 `tryAccept`（校验 accept 树 + 主线互斥）→ 步骤由 `objectives` 事实变化驱动推进 → 达成末步转 `COMPLETABLE` → `deliveryRule` 满足且 `tryClaim` → 走 §6.3 → `COMPLETED` + 解锁后续。

**异常恢复规则（原文完全缺失，必补）**：

| 场景 | 恢复规则 |
|---|---|
| 断线重连 | 读快照 `status+currentStepId+stepProgress+baseline_facts+stepBaselines+doneActionIds`；重放当前步骤 `onEnter` 中的 `REPLAYABLE` 动作、跳过已记录的 `ONCE` 动作；**不重置 stepBaselines**；幂等键防重复发奖；`COMPLETABLE` 保留 |
| 玩家死亡 | 不清任务/步骤进度；副本内按 `failRule`（默认原地复活/重置波次） |
| 退出副本 | 副本步骤记录"已进入"事实；中途退出回退到"未进入副本"步骤 onEnter，不判失败 |
| 任务物品丢失 | 关键道具标记不可丢弃/出售；仍丢失则读档按"应持有量"补发（附录 B P1） |
| 重复触发/连点 | 接取/交付/领奖均过幂等键；已 COMPLETED 一次性任务不可再接 |

### 4.4 事件公共信封【本版补：原文事件只有业务参数】

WAR3 的房间/副本/组队/镜像战会立刻依赖公共字段。**所有领域事件统一携带信封**：

```
EventEnvelope {
  playerId       // 事实归属玩家（组队/多人必需）
  scope          // 个人 / 房间 / 副本实例；决定事实写入哪个作用域
  instanceId?    // 副本/秘境实例 ID（1.2/1.4/1.8 镜像战按实例隔离）
  seq / ts       // 事件序号与时间戳（去重、乱序保护、离线批量结算排序）
  sourceEntity?  // 来源单位/NPC/掉落物（击杀归属、镜像战区分本体与分身）
} + 业务参数
```

约定：谓词 `resolveDeps` 与锚点键按需带 `scope`（如 `kill.monster.{id}@instance` vs `@personal`）；镜像战击杀只写副本实例作用域，不污染个人击杀事实。

**`seq / ts` 权威生成规则【本版补】**：`seq/ts` 必须由**主机/服务端权威生成**，按 `(playerId, scope)` 维度**单调递增**；**禁止**由客户端或单位本地时间生成。只有这样，击杀归属、断线补算、镜像/副本去重、离线批量结算排序才有唯一可信标准。乱序到达的事件按 `seq` 排序后再喂给事实层。

| 组合子 | `all/any/not` 递归 | ★ | V1 | 复合条件 | 境界+材料+灵石同时满足等 |

事件只带稳定 ID（monsterId/itemId/recipeId/npcId/dialogueNodeId/sceneId/realmId），不用中文显示名做逻辑条件。

#### 5.1 谓词版本归属与事件接口预留【本版补】

谓词分两半：**任务侧谓词实现**（写在任务系统内，成本低）与**领域侧事件广播**（要对应系统在完成点 `se.emit`）。二者版本可能不同——谓词能先写，但没有事件源就永远 UNKNOWN。规则：**谁的系统先做，谁负责在做的时候把事件接口一并埋好（哪怕当版任务还用不到）。**

| 谓词 | 依赖领域系统 | 系统版本 | 事件接口预留要求 |
|---|---|---|---|
| `cultivation.settled`/`realm.changed`/`breakthrough.finished` | 修炼/境界/突破 | V2 | V2 做这些系统时，结算点直接 emit，不留后患 |
| `alchemy.recipe_completed`/`gather.completed` | 炼丹/采集 | V4 | **V4 做炼丹系统时**在"炼成/采完"点 emit；V2 若已要 1.5 打坐，打坐结算事件属修炼系统同批做 |
| `skill.proficiency_changed` | 功法神通 | V5 | V5 做功法系统时在"熟练度变化"点 emit |
| `sect.contribution_changed` | 宗门 | V5 | V5 做宗门系统时在"贡献变化"点 emit |

结论：**任务系统不提前替别的系统做事件广播**；但每个系统的详设 checklist 里要写死"完成点必须 emit 对应领域事件"，由本表约束。V1 只依赖 V1 就绪的事件（combat/dialogue/region/item/quest），不依赖任何后置系统。

### 7.0 新手线跨版本落地策略【本版新增·修复断链与顺序矛盾】

**问题**：新手线数字序是 1.2→1.3→1.4→1.5→1.6→1.7→1.8，但依赖系统的版本序是 V1(1.2)→V4(1.3)→V5(1.4)→V2(1.5/1.7/1.8)。若坚持数字序上线，玩家在 V1 打完 1.2 后会卡在 1.3 炼丹（V4 才有炼丹系统），整条线要等到 V5 才连贯——这不可接受。同时 1.6 前置是 1.5(V2)，却在 V1 标"框架就绪"，也矛盾。

**裁定：新手线按"版本就绪顺序"上线，用『章节内排序』与数字 ID 解绑。** 玩家实际体验顺序随版本推进逐步补全，而非死守 1.2→1.8 的编号。落地顺序如下：

| 上线批次 | 版本 | 玩家实际可玩到的新手环节 | 编号 |
|---|---|---|---|
| 第 1 批 | **V1** | 试炼击杀（含背包/拾取/装备/吃丹）→ 委托能力**仅预留**（不上内容） | 1.2 |
| 第 2 批 | **V2** | 打坐修炼→授服饰→首次委托→收集突破材料→破境筑基 | 1.5→1.6→1.7→1.8 |
| 第 3 批 | **V4** | 首次炼丹（炼丹系统就绪后插入体验流） | 1.3 |
| 第 4 批 | **V5** | 长老授功法（功法系统就绪后插入体验流） | 1.4 |

**两条硬约束**：

1. **1.6 修正**：V1 只做"委托系统框架/任务栏能力预留"，**首次委托内容(1.6)与其前置 1.5 一起放 V2**。V1 不上任何可玩委托。（原文 V1"1.6 框架就绪"仅指引擎能力，本表明确内容归 V2，消除前置矛盾。）
2. **1.3 的 V2 跨版本依赖**：1.3 流程含"打坐调息约 10 秒"，依赖修炼系统(V2)。1.3 整体在 V4，此时 V2 修炼早已就绪，**无占位问题**。真正需要占位的是相反方向——若产品坚持让炼丹/功法在数字位插队到 V2 前体验，才需占位；本裁定按版本就绪排序，**不制造这种占位需求**。

**衔接体验**：第 1 批(V1)结束后，玩家在 1.2 交付后不是"卡住"，而是进入**开放练级/委托预留入口**；第 2 批(V2)上线时，1.5 起的主线以"陆青/孙玲再引导"自然接上。炼丹(1.3)/功法(1.4)后补时，作为"补授"剧情插入，不破坏已上线顺序。此策略需产品确认（附录 B P0-新增）。

#### 7.0.2 新账号 / 老账号 迁移双表【本版新增·编号解绑的上线落地】

编号解绑后，每次版本上线要回答"新号走什么顺序、老号如何补历史环节"。规则如下：

**新账号（该版本后新注册）**：直接走当前版本已开放的完整体验顺序，无补做问题。

| 上线版本 | 新账号可玩到的新手顺序 |
|---|---|
| V1 | 1.2 → 开放练级 |
| V2 | 1.2 → 1.5 → 1.6 → 1.7 → 1.8 → 开放 |
| V4 | 1.2 → 1.5 → 1.6 → 1.7 → 1.8 → **1.3(补授炼丹)** → 开放 |
| V5 | …1.8 → 1.3 → **1.4(补授功法)** → 开放 |

**老账号（版本上线前已存在）补偿**：

| 新上线环节 | 已越过该内容的老号处理 | 机制 |
|---|---|---|
| V4 加 1.3 炼丹 | 已筑基老号（早过了炼丹点）→ **不插回历史流程**，以"补授任务"形式在当前进度追加接取；完成即补发对应奖励/解锁 | 前置设为 `completed_quest(1.8)`，老号天然满足，登录后任务栏出现"补授：丹道初窥" |

| 任一环节 | 从未开始新手线的老号 | 按新账号顺序正常走 |

**存档兼容**：老号缺失的新手任务记录，读档时按"占位默认 NOT_ACCEPTED"补齐（依赖 V0 已预留的任务占位字段，见 §7.1）；已跨过的前置用 `completed_quest` 事实判定，不倒查历史。**此双表需产品确认（附录 B P0-新增）。**

#### 流程与操作

> 苦修有成，境界进阶，值得嘉许。修行之路漫漫长，突破不是结束，是新的开始，切莫心生自满，踏实砥砺方能大道长远。

**玩家：** "弟子时刻谨记。"

- 全屏金光特效
- 雷劫、环境光和天气恢复
- 世界播报："【玩家名】成功筑基，踏上仙途！"
- 境界变化、属性提升和新功能解锁提示

- 以第一章结束提示给予明确阶段成就感

#### 所需资源

- 陆青、周文扬陪同站位及环境演出

- 首次筑基成功率和失败逻辑需在上线前最终确认
