## 二、ConfigScene 场景配置表（第2个页签）

### 2.1 字段说明

| 字段名 | 中文名 | 说明 |
|--------|--------|------|
| config_id | 场景ID | 程序读取的唯一标识，从1开始顺序递增（1,2,3,4...） |
| scene_name | 场景名 | 中文名称，给策划/美术看的，如"银杏渔村"、"比奇城" |
| CameraDistance | 镜头距离 | 摄像机到角色的距离 |
| CameraAngle | 镜头初始角度 | 摄像机的初始旋转角度 |
| Minimap | 小地图 | 小地图资源文件名，如`mini_yxyc` |
| Bigmap | 大地图 | 大地图资源文件名，如`big_yxyc` |
| Grade | 场景等级 | 对应装备档次：0=新手、1=沃玛、2=祖玛、3=赤月... |

### 2.2 场景等级（Grade）定义

场景等级对应玩家的装备成长路线：

| 等级值 | 装备档次 | 典型场景 |
|--------|----------|----------|
| 0 | 新手 | 银杏渔村 |
| 1 | 沃玛 | 比奇城、兽人古墓 |
| 2 | 祖玛 | 尸魔洞、毒龙 |
| 3 | 赤月 | 石墓、未知暗殿 |
| 4 | 苍月 | 苍月岛 |

**设计原则**：场景等级跟随玩家的升级路线，玩家随着等级提升会按场景ID顺序进入更高等级的场景。

### 2.3 ID与名称的关系

- **程序只认ID**：程序读取的是`config_id`（1,2,3,4...）
- **人看名称**：策划和美术通过`scene_name`知道这是什么地方
- **示例**：
  - 场景ID=3 → 程序看到的是"3"
  - 场景名="兽人古墓第一层" → 人一看就知道这是兽人古墓

## 三、region_config 区域配置主表（第1个页签）

### 3.1 字段说明

本表有两个"ID"——两者用途不同，不可混淆：

| 字段 | 类型 | 给谁用 | 示例 |
|------|------|--------|------|
| **config_id**（区域ID） | 纯数字 | **程序**索引查表 | `70015101` |
| **scene_name**（地编名） | 文本标识 | **地编工具**中肉眼定位 | `biqi_N5_1_1` |

| 字段名 | 中文名 | 说明 |
|--------|--------|------|
| config_id | 区域ID | 程序唯一主键，通过公式自动生成（见3.5） |
| rect_name | 区域名 | 给策划看的区域名称，如"宗门-区域1" |
| 备注 | 备注 | 该区域的功能说明 |
| scene_name | 地编名 | 地编工具中的区域标识，通过公式自动生成（见3.3） |
| scene_id | 所属场景ID | 对应ConfigScene表的config_id。**注意**：原表头名"所属区域ID"不准确，此处实际为"所属场景ID" |
| 类型 | 类型 | 区域类型（如野怪、NPC、特效、镜头等） |
| 面积 | 面积 | 区域大小等级，用于估算刷怪数量 |
| 行 | 行号 | 区域在场景中的行位置 |
| 序号 | 序号 | 同一行内的序号 |
| is_main_rect | 是否场景主区域 | 标记主区域 |
| is_born_rect | 是否出生区域 | 标记玩家出生点 |
| teleport_target | 传送目标场景 | 传送点目标 |
| 刷怪方案说明 | 刷怪方案说明 | 文字说明 |
| RefreshMonsterPlan1 | 刷怪方案1 | 刷怪方案ID |
| RefreshMonsterPlan2 | 刷怪方案2 | 备用刷怪方案ID |

### 3.2 核心概念：区域划分

**什么是区域？**

一个大的场景（如"宗门"）会被划分为多个小区域（十几个到二十几个），每个区域是一个独立的刷怪/功能区块。

**区域的作用：**
1. **刷怪控制**：每个区域独立配置刷怪方案
2. **功能分区**：不同区域承担不同功能（战斗区、NPC区、休息区等）
3. **面积估算**：根据区域面积大小决定刷怪密度

### 3.3 面积、行、序号的含义

这三个字段用于**地编命名**和**刷怪配置**：

| 字段 | 作用 |
|------|------|
| 面积 | 区域大小等级（如5=大面积，1=小面积）。大面积区域刷20个怪，小面积刷5个怪 |
| 行 | 区域在场景中的行位置。看地图就知道这个区域属于场景的哪一块（入口第1行、深处第3行...） |
| 序号 | 同一行内的序号，用于区分同一行的不同区域 |

**地编名生成公式**：
```
data_name = 场景前缀 + 类型缩写 + 面积 + "_" + 行 + "_" + 序号
```

示例：`biqi_N5_1_1` 表示比奇城野怪区，面积等级5，第1行第1个区域

> **为什么行和序号能反推位置？** 因为区域划分时就是按地图从上到下、从左到右分的行，第几行=地图上的第几排。光看"行=3"就知道是场景深处。

### 3.4 所属场景ID（scene_id）

- 这个字段对应ConfigScene表的`config_id`
- **命名建议**：原字段名"所属区域ID"不准确，建议改为"所属场景ID"
- 示例：
  - scene_id=1 → 银杏渔村
  - scene_id=2 → 比奇城
  - scene_id=3 → 兽人古墓第一层

### 3.5 区域ID生成公式与解码

**公式**：
```
config_id = 70000000 + scene_id × 10000 + 面积 × 1000 + 行 × 100 + 序号
```

**各数位含义**：

| 位 | 权重 | 含义 | 示例 |
|----|------|------|------|
| 千万位 | 7000xxxx | 固定前缀 | 7 |
| 百万位~万位 | xxxx | 场景ID | 1 = 银杏渔村 |
| 千位 | x000 | **面积**等级 | 5 = 最大 |
| 百位 | 0x00 | **行号** | 1 = 第一排 |
| 十位 | 00x0 | 预留（始终为0） | 0 |
| 个位 | 000x | **序号** | 1 = 该行第一个 |

**示例解码**：
- `70015101` → 场景1、面积5、行1、序号1
- 配刷怪的人看到这个ID就知道：**场景入口行最大的区域，适合放密集刷怪**

### 3.6 为什么要这样编码 —— 设计意图

这套编码的核心价值不是"好看"，而是**减少上下文切换**。

**场景**：G怪物刷新配置表里引用了这个区域ID，配刷怪的人正在填怪物刷新配置。

**如果ID是自增数字（1,2,3...）**：
1. 看到"区域3"
2. 切到地图 → 确认区域3的位置和大小
3. 切回来 → 决定刷多少怪
4. 每配一个区域都要切屏一次

**现在ID编码了空间信息（70015101）**：
1. 看到 `70015101`
2. 千位=5 → 大区域，多放怪
3. 百位=1 → 入口行，放引导性怪物
4. 不用切屏，直接在同一页填完

**信息就在ID里，不用查地图。** 这就是为什么编码设计比字段名重要。

## 五、与怪物刷新配置表的联动

场景区域配置必须结合 `G怪物刷新配置.xlsx` 理解其设计意图。

### 5.1 两个表的链接点

region_config 的 **RefreshMonsterPlan1** 字段引用 spawn_config 的 config_id：

```
区域配置表（region_config）              怪物刷新配置（spawn_config）
┌──────────────────────┐              ┌──────────────────────────┐
│ config_id: 70015101  │  引用        │ config_id: 1001302       │
│ 场景=1, 面积=5       │──────────→   │ 场景=银杏渔村             │
│ 行=1, 序号=1         │ Refresh      │ 类型=BOSS                │
│ 刷怪方案1: 1001302   │ Monster      │ Monster1=60011000 × 1    │
│                      │ Plan1        │ Monster2=60011001 × 2    │
│                      │              │ Monster3=60011003 × 3    │
└──────────────────────┘              └──────────────────────────┘
```

### 5.2 配刷怪时无需看地图

配刷怪的人操作流程：

1. 在spawn_config中创建刷怪方案（有哪些怪物、各几只）
2. 回到region_config，找到引用该方案的区域行
3. 看区域的 config_id：`70015101`
   - 面积=5 → 全图最大区域 → 多放怪、高刷新频率
   - 行=1 → 场景入口 → 放低难度怪作为引导
   - 类型=野怪 → 战斗区，不放NPC
4. 据此调整 spawn_config 中的怪物种类和数量

**全程不用打开地图编辑器**。配置人的设计意图（"宗门入口放弱怪"）在画区域时就已经定了，配表时只需要从ID解码出位置信息即可。

### 5.3 怪物刷新配置表 字段说明

| 字段 | 中文名 | 说明 |
|------|--------|------|
| config_id | 刷怪方案ID | 程序唯一主键，公式自动生成（见5.5） |
| 场景名 | 场景名 | 对应哪个场景（银杏渔村、比奇城...） |
| 类型 | 刷怪类型 | BOSS / 精英 / 小怪0~7 |
| 序号 | 序号 | 同场景同类型下的第几个方案 |
| 说明 | 说明 | 方案描述，如"鸡、鹿"、"半兽人2" |
| position | 刷怪位置 | 随机点/固定点等 |
| MonsterID1~10 | 怪物ID | 该方案最多配10种怪物 |
| MonsterCount1~10 | 怪物数量 | 每种怪物各几只 |

### 5.4 刷怪方案的设计思路 —— ID编码即方案类型

怪物刷新配置表的设计原则和场景区域一致：**ID编码自身信息**。

每个 `config_id` 是一个**完整的刷怪方案**，一行就定死了"哪个场景、哪种档次（BOSS/精英/小怪）、有哪些怪物、各几只"。

**ID公式**：
```
config_id = 1000000 + 场景代码 × 1000 + 类型代码 × 100 + 序号
```

**各数位含义**：

| 位 | 含义 | 示例值 | 解码 |
|----|------|--------|------|
| 百万位 | 固定前缀 | 1 |
| 万~十万位 | 场景代码 | 001 = 银杏渔村 |
| 千位 | 保留（0） | 0 |
| 百位 | **类型代码** | 0=BOSS、1=精英、2=小怪0... |
| 十位~个位 | **序号** | 01 = 该场景该类型第1个方案 |

**类型代码映射**：

| 代码 | 类型 |
|------|------|
| 0 | BOSS |
| 1 | 精英 |
| 2 | 小怪0 |
| 3 | 小怪1 |
| ... | 小怪2~7 |

**看到ID就能判断方案档次**：`1001302` → 场景1（银杏渔村）、类型0（BOSS）、序号2 → 银杏渔村第2个BOSS方案。

配怪的人不用看"类型"那一列，光看ID就知道这是刷BOSS还是刷小怪。

### 5.5 两个表联动的完整流程

```
场景设计阶段：
  策划划分场景区域 → 填 region_config
    ├ config_id: 70015101（场景1 面积5 行1 序号1）
    └ 类型=野怪 → 知道这是战斗区

↓ 区域ID已编码"这是场景入口最大区域"

刷怪配置阶段：
  策划创建刷怪方案 → 填 spawn_config
    ├ config_id: 1001302（场景1 BOSS 序号2）
    ├ Monster1=半兽人统领 × 1
    └ Monster2=半兽人 × 3

↓ 方案ID已编码"这是银杏渔村BOSS方案"

关联阶段：
  region_config.RefreshMonsterPlan1 = 1001302
  → 配对完成：入口大区域 × BOSS方案 = 场景入口BOSS战

配表人根本不需要打开地图——
  区域ID告诉他位置和大小，
  方案ID告诉他档次和内容，
  两个编码加在一起就能完成配对决策。
```

## 六、地编刷区域操作流程

### 6.1 策划设计阶段

1. **划分场景**：确定一个大场景要分成多少个区域
2. **定义功能**：每个区域是干什么的（刷怪区、NPC区、休息区等）
3. **估算面积**：给每个区域一个面积等级（1-5）
4. **确定位置**：确定每个区域在场景中的行位置和序号

### 6.2 表格填写

1. 在ConfigScene表中添加场景基础信息
2. 在region_config表中添加区域信息：
   - 填写`所属场景ID`（对应ConfigScene的config_id）
   - 填写`区域名`（给自己看的）
   - 填写`类型`、`面积`、`行`、`序号`
   - 地编名和区域ID通过公式自动生成
3. 填写刷怪方案ID到`RefreshMonsterPlan1`

### 6.3 地编操作

1. 打开地编工具
2. 根据表格中的`scene_name`（地编名）找到对应区域
3. 在场景中框选对应大小的区域
4. 关联刷怪方案

## 八、注意事项

1. **程序只读ID**：ConfigScene的`config_id`和region_config的`config_id`是程序唯一识别的字段
2. **人看名称**：`scene_name`和`rect_name`是给策划/美术看的
3. **公式字段不要手动修改**：区域ID和地编名是通过公式自动生成的
4. **刷怪方案**：`RefreshMonsterPlan1`关联到刷怪配置表
5. **枚举表维护**：新增场景时需要在枚举表中添加对应的地编前缀
