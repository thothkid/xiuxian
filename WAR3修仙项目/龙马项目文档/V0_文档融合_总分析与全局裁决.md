## 二、全局口径裁决（最高优先级 · 已定案）

> 这 8 条是跨文档地基。不先定死，25 份会各搞各的（现状就是如此）。每条给：两版做法 → 裁决 → 理由 → 影响范围。

### G1 怪物命名与 ID 体系
- [事实] 小龙：灵兔/石魔/狼妖，主键 `config_id` = 1/2/3。
- [事实] 小马：小野狼/石傀儡/练气妖兽，主键 `monster_id` = 1001/1002/1003。
- **裁决**：采用**小马**的命名（小野狼/石傀儡/练气妖兽）+ `monster_id` 1001/1002/1003。`【否决点·命名是纯取舍】`
- [理由] ①"练气妖兽"锚定境界词汇（练气），与世界观体系一致；三名都直给难度直觉。②`monster_id` 分段编码（1xxx=怪物段）与物品段编码同构、可扩展；`config_id 1/2/3` 太通用，多表主键全叫 config_id 易混。③小马怪物表字段更全（model_scale）。
- **影响**：P05、P07、P09、P18、P19、P25 全部统一到此命名+ID。

### G2 物品命名与 ID + 关键数值
- [事实] 小马 10 种，item_id 分段 1001/2001/3001/4001，`item_type` 用 int(1-5)，铁剑 atk+3 / 布衣 def+2，回灵丹 MP+50，灵石堆叠 999。含灵泉水(双恢复)、妖兽内丹(堆叠上限10)。
- [事实] 小龙 8 种，item_id 顺序 1-8，`type` 用字符串枚举(CURRENCY/CONSUMABLE/MATERIAL/EQUIP)+`sub_type`，铁剑 atk+5 / 布衣 def+3，回灵丹 MP+30，灵石堆叠 9999。无灵泉水、无妖兽内丹。
- **裁决**：物品集用**小马的 10 种**、item_id 分段编码；枚举字段用**小龙的字符串枚举风格**（type 大类 + func_type 功能，废弃 int 编码）；其余字段（sell_price/quality/level_require/description）取小马的完整集。装备数值取**小马 +3/+2**，回灵丹取 **MP+50**，灵石堆叠 **999**。`【否决点·装备数值±可调】`
- [理由] ①10 种比 8 种多覆盖两条闭环分支：灵泉水验"双恢复"、妖兽内丹(堆叠10)验"低上限满包提示"——验收价值高（克制让位于可测性）。②字符串枚举可读性强，配置里 "WEAPON" 比 "4" 不易错，程序读 enum 字符串无额外成本——这条小龙更优。③装备 +3/+2 与小马的装备文档/战斗示例内部自洽(atk10→13)；+5/+3 也能用，差异在 13/15，留作 thoth 可调。④回灵丹 MP+50 与聚气丹 HP+50 对称。
- **影响**：P10、P09、P12、P14、P16、P24。

### G3 八张配置表 · 表名与命名规范
- [事实] 小马：`attr_config / init_char_config / linggen_config / monster_config / spawn_config / drop_config / item_config / region_config`（统一 `_config` 后缀）。
- [事实] 小龙：`attr_config / init_character / spirit_root / monster / monster_spawn / drop_table / item / map_zone`（简名，仅 attr 一致）。
- **裁决**：全部采用**小马的 `{domain}_config` 后缀**统一命名。
- [理由] ①统一后缀让 `ConfigManager.get_table("xxx_config")` 一致；②`monster`/`item` 这种裸名太通用，易和代码里的类名/变量撞，加后缀消歧。字段名两版都 snake_case，保留；每表主键用域前缀（monster_id/item_id/region_id），不用通用 config_id。
- **影响**：P02、P05、P07、P09、P10、P14、P15、P17、P18 全部表名归一。

### G4 区域体系（ID / 字段 / 区域集）
- [事实] 小马：5 区，id 0-4，出生区(0)⊂安全区(1)嵌套 + 刷怪区A(2)/刷怪区B(3) + 边界(4)；字段 left/top/right/bottom；含完整 RegionManager 伪代码。
- [事实] 小龙：5 区，id 1-5，安全区(1)+刷怪区(2,一大块)+掉落测试区(3)+边界西(4)/边界东(5)；字段 rect_left/...；枚举 SAFE/COMBAT/LOOT_TEST/BOUNDARY。
- **裁决**：以**小马的区域框架为底**（出生区⊂安全区嵌套、刷怪区A/B、程序模型完整），**并入小龙的"掉落测试区"作为 region_id=5**。`region_id` 0 起、字段用小马的裸 left/top/right/bottom。`【否决点·要不要保留刷怪区A/B拆分】`
- [理由] ①小马的嵌套出生区解决"出生瞬间不在战斗中"，程序伪代码齐全（可执行性高）；②小龙的"掉落测试区"(空地人工放怪)对验收拾取/背包确有用，低成本并入；③A/B 拆分与刷新点精确放置配合(见 G5)。
- **影响**：P18、P07、P06、P14。

### G5 刷新模型
- [事实] 小马：region 驱动，字段 region_id/monster_id/count/max_alive/init_delay/respawn_interval/spawn_type=interval，坐标交给区域表。
- [事实] 小龙：坐标驱动，字段含 center_x/center_y/radius + spawn_type FIXED_POINT/AREA_RANDOM，三刷新点精确(灵兔坡800,0 / 石魔谷1200,0 / 狼妖林1600,0)。
- **裁决**：字段命名用**小马**(max_alive/init_delay/respawn_interval)，刷新定位**并入小龙的 center_x/center_y/radius + AREA_RANDOM**。即 spawn 既带 region_id(分类/脱战边界)又带精确刷新圆。
- [理由] V0 三只怪要落在一个大刷怪区内的三个不同点，小龙的圆心+半径比"整个区域矩形随机"精确；小马字段命名更规整。两者并入无矛盾(region_id 管归属与脱战，center/radius 管落点)。
- **影响**：P07、P18。

### G6 验收清单格式
- [事实] 小马：编号式 `[Sxx-xx]`，按程序系统 S01-S20 分组，含"一票否决项"(S07-02 严禁力量/敏捷/智力)，索引称约 135 条。
- [事实] 小龙：表格式，按功能(房间/创角/属性/怪物/AI/战斗/掉落/背包/装备/存档/异常/冒烟)分 12 节，61 项，含 操作步骤/预期结果/✓ 三列 + 末尾通过率结论表(冒烟100%+其余≥95%)。
- **裁决**：骨架用**小马编号式([Sxx-xx]绑系统)**，列结构并入**小龙的 操作步骤/预期结果/勾选**三列，末尾并入**小龙的通过率门槛表**。
- [理由] 编号绑系统利于程序自检与回溯；小龙的操作/预期列让非程序也能照着验；小龙的通过率门槛是明确的 V0 完成线。两者互补。
- **影响**：P19。

### G7 提示文案体系
- [事实] 小马：PROMPT_ID 枚举(Lua)+ID码(A/B/C/D/E段)+占位符替换规则+颜色/时长/堆叠/动画/音频全规格，文案修仙化(道基已固/储物袋已满)。
- [事实] 小龙：分类表(系统/战斗/物品/角色)，文案直白(保存成功/背包已满)，规格较简。
- **裁决**：结构采用**小马**(枚举+ID码+全规格，程序可直接接)；**文案风格层级交 thoth 定**——修仙化(道基已固)更有味，直白(保存成功)对灰盒测试更清楚。`【否决点·文案修仙化程度】`
- [理由] 小马结构完整度碾压；唯文案"道基已固"对 V0 灰盒阶段的可读性不如"保存成功"，这是体验取向，不该我替你定死。
- **影响**：P24、P21、P22。

### G8 文档元信息规范
- [事实] 小马每份带 生成者/依据文档/关联文档/下一步/状态行；小龙仅 版本/对应/依赖。
- **裁决**：融合版统一采用**小马的富元信息**(依据/关联/下一步)。
- [理由] 利于后续维护与追溯。低成本。
